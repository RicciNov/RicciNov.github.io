类与对象

By Ricci_April

Published 2016-08-13

Python中的类与对象

关于类与对象，自然是编程模式的一种——面向对象（Object Oriented, OO）中的主要特点元素。作为学java入门编程的，对这个更是由衷地感到熟悉亲切。世界万物都是对象，而所有对象都由类实例化来的。搭建解空间、程序设计，均是以类作为基础元素。
然而Python中的类与对象还是有些特别，也许这就是解释型动态语言与静态语言的区别所带来的实质性感受。

不系统的讲解类与对象，而是针对其与静态语言（主要是Java）中不同的地方展开。我觉得主要从类属性与实例属性、绑定概念、多重继承三个方面叙述，另外类中的方法涉及诸多，包括定义方式、解释器默认地特殊方法等，也会掺杂其中。

类属性与实例属性

这里首先要强调的有如下两句话，

类和实例都是命名空间（namespace），类是类属性的命名空间，实例则是实例属性的命名空间。

还有就是：

如果说类是一种数据结构定义类型，那么实例则声明了一个这种类型的变量。

我觉得，如果能够仔细理解了如上两句话，基本上就明白了类与实例之间剪不断理还乱的关系。不是计算机科班出身，也没有系统性地研究过计算机相关知识，但是通过看过的听过的掌握的，认为所谓的程序语言其实就是在处理计算机内存的问题，内存地使用涉及到了成本和效率，如何在有限成本下高效利用内存，其反应到了上层就是计算机解决问题的能力和速度。当前存在如此多的程序语言，其设计创建的初衷也肯定是因为在某些特定环境下，前面的语言不好使，后面的语言才出现了。数据结构的设计同样如此。

类是一种程序员可以自定义的数据类型，是构建解决问题空间的基本元素，而实例就是使用该类型创建出的对象，对应实际问题中的各个角色。这也能说明了两者的关系，实例是有类衍生出来的，在强类型语言中，类中定义了什么属性（数据和函数），实例中才能拥有什么属性，不可能自己创造，由于属性值不同才造成了实例的不同（当然这样说，有些函数式编程语言的特点，准确说，可能要说引用不同）；而弱类型语言中，实例或者内建类型的变量，可以在运行时自行改变属性方法等。在这种情况下，Python应该是弱类型语言了。因为，Python中的实例，可以自行添加属性，这些属性与类中的属性其实是没有关系的。这也直接导致了实例与类的关系，不在是上下层关系，类似于平行关系了。

也就是从另一个角度看，类与实例都是命名空间，类是类属性的命名空间，实例是实例属性的命名空间；两者并行存在着，其中能连接两者的，貌似也就是实例与类的对应关系了。

理解了上述，其实关于类数据属性、实例数据属性，很清晰就可以看出其存在的意义，而这也是与Java语言的不同点之一。Java中同一个类实例化出来的所有对象，都拥有同样的属性类型（重载、泛化暂不谈），而Python中的同一个类实例化出来的不同对象，可不一定了，因为每个实例都自主拥有自行定义实例属性的权利和能力了。

类空间和实例空间的并行，有点类似于静态数据和动态数据。

绑定关系

绑定（binding）的存在，大概觉得应该是和底层数据的组建关系有关，但是为什么其他语言没有呢，貌似这也是Python语言有意思的地方。

绑定关系，指的是，在Python中，所有类方法的调用，必须要有调用者，这个调用者不能是Python解释器，当然实际上的调用中介肯定是解释器，只是在某个方法被调用的时候，必须要标明到底是哪个实例或者类在调用它，那么把这个调用者（实例或者类）与方法联系起来就是绑定。

另外，方法仅仅是类内部定义的函数（这意味着方法是类属性而非实例属性），一般方法被调用是必须要在类实例化后，将实例传递给方法，也就是绑定后；否则方法是不能被调用的。当然，现在还有一种静态方法和类方法，可以直接由类调用，当然也可以由实例调用，并没有任何不同。

这也是方法中形参的第一个通常是self的原因，其代表了接收到的绑定实例；在由实例调用方法时，可以省去手工传递该参数，解释器会帮我们完成将该实例传递给self的动作。

多重继承

多重继承本来是比较简单的事情，但是困难地是，如果在子类有多个父类时，其可能继承多个相同或者类似的方法，因为在子类重载方法时，必须要先对父类的方法做点儿什么，那如何在多个父类中找到该方法呢，这里就存在着一个算法，即方法解释顺序（MRO），其是解释器内部的逻辑。
简单点儿，我们要做的就是使用super(type[, obj])来有解释器自行找到子类的父类，然后调用相应方法。
这里针对多重继承不再多累述，后续需要专门研究下。但是对super(type[, obj])再具体介绍下，毕竟这个是切实关系到使用的。这个方法返回参数type的父类。如果你希望父类被绑定，可以传入obj参数，但是这个obj必须是type类型的（或者其子类），否则父类不会被绑定。obj参数也可以是一个类，但它应当是type的一个子类。通常，当给出obj时：

  • 如果obj是一个实例，isinstance(obj, type)就必须返回True；
  • 如果obj是一个类或者类型，issubclass(obj, type)就必须返回True；

最后，属性_mro_包含了子类的所有的父类，并按照查找顺序排列。事实上，super()方法也就是在_mro_中包含的父类中查找，并由接收到的参数来界定查找起点等。

By Ricci_April

Published 2016-08-03

Dictionary

字典不过就是Map而已，但即使如此，也可以单独成一章吧，一小章。还没有研究它与其他语言中的Map有啥不同，暂且作为一个整理性质的初步学习。

某种程度上将，字典是列表的另一种形式，应该说是向上扩展了，更加general。这也构成了Dictionary的某些特点，比如：

  • 在列表中，索引必须是整数；但在字典中，它们可以是（几乎）任何类型；只要这些索引是可Hash的；
  • 字典中包含了一个索引的集合，即键Key，和一个值的集合，即值Value；一个键对应一个值，这种一一对应的关联被称为键值对，即key-value pair，也称为项Item；这种映射关系，正是字典的魅力所在；
  • 字典的某个键值如：eng2sp['one'] = 'uno'，正好看出，其类似于列表，不过索引值不是默认的依次排开的整数，而是自由定义的类型；而字典正是这些键值对的集合，如eng2sp = {'one' : 'uno', 'two' : 'dos', 'three' : 'tres'}
  • 与列表不同的是，字典中的键值对是无顺序保存的，而不像列表中的按索引大小排序；其中in运算符同样适用于字典，它可以用来检验字典中是否存在某键；in运算符对于列表和字典采用不同的算法。对于列表，它按顺序依次查找目标，随着列表的增长，搜索时间成正比增长；而字典不同，字典中的元素是使用哈希表Hashtable的算法保存的，所以在用in运算符查找时，无论字典中与多少，其搜索所需的时间都是一样的，这也正是Hashtable算法的牛逼之处。
  • 由于字典中的Key集合和Values集合拿出来后都可以作为list在管理编辑，因此针对字典的增删修选操作和衍生操作都没什么特别好介绍的，基本使用的时候再写个简单的函数操作；这里插个小片段，因为字典中的元素是无序的，如果要以确定的顺序便利字典，可以使用内建方法sorted()，例如：

其实貌似字典和列表的跨度很大一样，毕竟一个有序一个无序，但是想象其他语言中在list和map中，貌似还有个set无序集合作为桥接，以至于不太突兀。而在Python中好像没有。

UUID

UUID的全称，Universally Unique Identifier，通用唯一识别码。这个很清楚，凡是编程过的人一般都会立刻嘴角上扬了。毕竟在实际代码工程中，要给某个对象也好、线程也好、或者其它，使用一个唯一的、不会重复的ID，很重要，就像数据库中的主键。目前最广泛应用的UUID，就是微软的GUID(Globally Unique Identifier)。

UUID的生成规范定义的算法主要目的就是要保证其唯一性，当然这个唯一性是有限的，只有在特定的范围内才能得到保证，这和UUID的类型有关。后面细说。

UUID是16字节（1Byte = 8bits）128位长的数字，通常以36位长度的16进制数表示，示例如下：

3F2504E0-4F89-11D3-9A0C-0305E82C3301

其中字母是16进制表示，大小写无关。

具体来介绍UUID的内涵，可以从如下几个方面展开：

  • 经由一定的算法生成
    为了保证UUID的唯一性，规范定义了包括网卡Mac地址、时间戳、名字空间(Namespace)、随机或伪随机数、时序等元素，以及以这些元素生成UUID的算法。UUID的复杂特性也保证了其唯一性的同时，意味着只能由计算机生成。
  • 非人工指定，非人工识别
    UUID不能也不可能是人工指定的，除非冒着重复ID的风险。UUID的复杂性也决定了无法简单地分辨出UUID和对象之间的映射关系。
  • 在指定的范围内重复的可能性极小
    UUID的生成规范定义的算法主要目的是要保证其唯一性，但这个唯一性也是有限的，从特定的名字空间到全球。只有在特定的范围内才能真正保证其唯一性。

UUID有多个版本，每个版本的规范算法不同，应用范围也不同。
首先有一个特例，也就是Ni1 UUID，通常不会用到，其全部由0组成，如下：

00000000-0000-0000-0000-000000000000

  • Version 1：基于时间的UUID
    基于时间的UUID通过计算当前时间戳、随机数和机器Mac地址得到。由于算法中使用了Mac地址，这个版本的UUID可以保证在全球范围的唯一性；当然，也同样带来了安全问题。如果应用只是在局域网内使用，也可以考虑退化的算法，以IP地址代替Mac地址（Java的UUID往往是这样实现的，当然也是考虑了获取Mac的难度）。
  • Version 2：DCE安全的UUID
    DCE(Distributed Computing Environment)安全的UUID和基于时间的UUID算法相同，但会把时间戳的前4位置换成POSIX的UID或GID。这个版本的UUID在实际中应用较少。
  • Version 3：基于名字的UUID（MD5）
    基于名字的UUDI通过计算名字和名字空间的MD5散列值得到。这个版本的UUID保证了：相同名字空间中不同名字生成的UUID的唯一性；不同名字空间中的UUID的唯一性；相同名字空间中相同名字的UUID重复生成是相同的。
  • Version 4：随机UUID
    根据随机数，或者伪随机数生成UUID。这种UUID产生重复的概率是可以计算出来的。
  • Version 5：基于名字的UUID（SHA1）
    和版本3的UUID算法类似，只是散列值计算使用SHA1（Secure Hash Algorithm 1）算法。

从UUID的不同版本可以看出，Version 1/2适合应用于分布式计算环境下，具有高度的唯一性；Version 3/5适合于一定范围内名字唯一，且需要或可能会重复生成UUID的环境下；至于Version 4，一般不建议使用。
通常在标识对象或持久化数据时，推荐使用UUID作为唯一标识；但是以下情况不建议使用UUID：

  • 映射类型的对象，比如只有代码及名称的代码表。
  • 人工维护的非系统生成对象，比如系统中的部分基础数据。

对于具有名称不会重复的自然特性的对象，建议使用Version3/5的UUID，比如系统中的用户。如果用户的UUID使用Version 1的，那么如果在删除该用户后再次重建时，就很不会再生成同样的UUID，相当于人还是那个人，但是用户已经不是那个用户了。
